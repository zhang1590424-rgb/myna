- 文档版本：v1.0

- 范围：本地小白模型微调工具 M1 里程碑收尾（把已验证的真实训练接入产品闭环）
- 关联文档：`docs/superpowers/specs/2026-06-18-本地小白模型微调工具-design.md`（总策划案）、`AGENTS.md`（协作规则与交接）

## 1. 背景与目标

### 1.1 现状

M0 技术验证已通过：Mac（Apple M2 / 16GB / MPS）上用 LLaMA-Factory + LoRA SFT 把 Qwen2.5-0.5B 微调出了肉眼可见的效果（自我认知改写）。

M1 产品闭环骨架已搭好：前端五步向导、本地 FastAPI 服务、数据校验、训练配置生成器都已可用，但**“训练”这一步走的是 `MockTrainingEngine`（模拟）**——进度、loss、前后对比、导出全是假数据。

### 1.2 目标

把 M0 已验证的真实训练能力，接进 M1 现有的产品流程，替换掉模拟部分。完成后：

- 小白点完五步向导，**真实启动一次本地 LoRA SFT 训练**；
- 进度条、loss 曲线反映**真实训练状态**；
- 训练前后对比展示**模型真实回答**（base vs base+LoRA）；
- 导出得到**可用的 LoRA adapter**。

### 1.3 非目标（本期不做）

- 多模型自动下载（属于 M3）。
- 桌面打包 Tauri/Electron（属于 M2）。
- DPO/PPO 等强化学习（Mac 跑不动，后续）。
- LoRA 合并导出为完整模型（本期只导出 adapter 目录）。

## 2. 设计原则

沿用 `AGENTS.md`：

1. **产品层与引擎隔离**：前端不拼训练参数；后端通过适配层生成 LLaMA-Factory 配置并调用 CLI。
2. **不伪装真实训练**：真实引擎与 Mock 引擎并存，且必须能明确区分；默认走真实引擎，环境不满足时显式降级到 Mock。
3. **小白优先**：所有用户可见文案说人话；专业参数与失败原因都翻译成可理解的提示。
4. **不魔改上游**：`LLaMA-Factory/` 视为第三方依赖，仅通过 `llamafactory-cli` 调用。

## 3. 总体架构

```
前端 web/  ──/api/*──>  FastAPI (main.py)
                            │
                            ├── DatasetStore        读用户数据
                            ├── LlamaFactoryConfigBuilder  生成真实训练配置
                            ├── TrainingEngine（接口）
                            │     ├── LlamaFactoryTrainingEngine  真实训练（默认）
                            │     └── MockTrainingEngine          演示/降级
                            ├── hardware.py         设备与精度选择
                            └── infer.py            真实前后对比推理
                                    │
                                    └── 子进程 llamafactory-cli train <config>
                                            │ 写 trainer_log.jsonl
                                            └── 输出 LoRA adapter 到 run_dir/output
```

关键变化：

- 抽出 **`TrainingEngine` 抽象接口**，`Mock` 和真实引擎都实现它，`main.py` 通过工厂按环境选择。
- 真实训练用 **子进程**跑 `llamafactory-cli`，而不是在 web 进程内训练（避免阻塞事件循环、便于安全终止）。
- 进度通过**轮询解析 `trainer_log.jsonl`** 获得，前端轮询接口不变。

## 4. 组件设计

### 4.1 `hardware.py`（新增）— 设备与精度

集中解决 M0 踩过的“MPS 必须 fp32”坑，避免散落在各处。

| 函数 | 职责 |
|---|---|
| `detect_device() -> "mps" \| "cuda" \| "cpu"` | 探测可用加速设备 |
| `select_precision(device) -> {"bf16": bool, "fp16": bool}` | MPS/CPU 返回全 False（fp32）；CUDA 返回 bf16 |
| `training_env() -> dict[str,str]` | 返回需要注入子进程的环境变量，含 `PYTORCH_ENABLE_MPS_FALLBACK=1` |

### 4.2 `llamafactory.py`（修改）— 配置生成

- 去掉写死的 `bf16: True`，改为根据 `hardware.select_precision()` 注入 `bf16`/`fp16`。
- `dataset_dir` 已是绝对路径（`str(dataset_dir)`），符合 M0 经验，保留。
- 其余字段（stage/sft、lora、template=qwen 等）不变。

### 4.3 `infer.py`（新增）— 真实前后对比

| 函数 | 职责 |
|---|---|
| `compare(base_path, adapter_dir, prompt) -> {"before","after"}` | 加载 base 模型答一次；挂上 LoRA adapter 再答一次 |

实现要点（复用 M0 `m0_compare.py` 已验证逻辑）：

- 用 `transformers` + `peft.PeftModel`；
- 设备由 `hardware.detect_device()` 决定，dtype 用 fp32（MPS）；
- `do_sample=False` 保证可复现；`max_new_tokens` 控制在 ~80，控制耗时。
- 推理较慢（Mac 上每次数秒到十几秒），在子进程或线程池中执行，避免阻塞事件循环。

### 4.4 `engine.py`（重写）— 引擎抽象 + 真实引擎

定义协议：

```python
class TrainingEngine(Protocol):
    name: str
    async def start(self, job: TrainingJob) -> None: ...
    async def stop(self, job_id: str) -> TrainingJob: ...
    async def compare(self, job_id: str, prompt: str) -> CompareResponse: ...
    async def export(self, job_id: str) -> ExportResult: ...
```

`JobStore` 保持不变（内存态，已够用）。

`MockTrainingEngine` 保留，作为降级/演示，行为不变。

新增 `LlamaFactoryTrainingEngine`：

- `start`：
  1. 读 `DatasetStore` 数据 + 模板 + 模型；
  2. 用 `LlamaFactoryConfigBuilder.prepare()` 生成 `train.yaml` 和数据；
  3. `asyncio.create_subprocess_exec("llamafactory-cli","train",config)`，注入 `hardware.training_env()`；
  4. 记录 `pid`、`run_dir`、`output_dir`、`trainer_log` 路径到 job；
  5. 后台 `asyncio.Task` 轮询 `trainer_log.jsonl`，回填 `progress / loss / message / learned_count`；进程退出后判定 `completed / failed`，并把 `output_dir` 指向真实 adapter。
- `stop`：向子进程发送终止信号（先 terminate，超时再 kill），状态置 `stopped`。
- `compare`：调用 `infer.compare()`，base 模型路径取自模型清单，adapter 取 job 的 `output_dir`。
- `export`：把 adapter 目录打包成 zip 返回。

#### 进度映射

`trainer_log.jsonl` 每行含 `current_steps / total_steps / loss / epoch / percentage`。映射：

- `progress = percentage`（封顶 99，进程退出且成功才置 100）；
- `loss` 追加到列表（前端画曲线）；
- `learned_count`：按 `percentage` 估算 `dataset_count * percentage/100`（仅展示用）；
- `message`：`正在学习第 N 轮，共 M 轮`。

#### 状态机

```
pending ──start──> running ──(进程正常退出且有 adapter)──> completed
                      │
                      ├──stop──> stopping ──(进程结束)──> stopped
                      └──(进程非零退出/无 adapter/异常)──> failed
```

### 4.5 `main.py`（修改）— 工厂与导出

- 启动时通过 `build_engine()` 选择引擎：环境就绪（`llamafactory` 可导入 + 有本地模型）→ 真实引擎；否则 Mock，并在 `/api/environment` 暴露当前引擎名，让前端/用户知道是不是真训练。
- `/api/compare`、`/api/training/jobs/{id}/export` 改为走引擎统一接口。
- 导出接口从返回 JSON 改为返回 **zip 文件流**（真实 adapter 目录）。Mock 引擎导出仍返回带说明的 zip/JSON，明确标注是模拟。

### 4.6 `domain.py`（小改）

`TrainingJob` 增加可选字段：`engine: str`、`pid: int | None`、`run_dir: str | None`、`eta: str | None`（剩余时间，给小白看的人话）。不破坏现有前端。

### 4.7 前端（最小改动）

现有轮询、loss 曲线、对比、导出逻辑已通用，**基本不用动**。仅：

- 准备环境页展示当前引擎模式（真实 / 演示），避免用户误会。
- 导出按钮已是 `window.location.href` 直链，zip 流天然兼容。

## 6. 错误处理（说人话）

| 真实失败 | 用户提示 |
|---|---|
| `llamafactory-cli` 不存在 | “训练组件还没装好，请先在准备环境页检查。” |
| 模型路径缺失 | “没找到本地模型，请先准备模型。” |
| 子进程非零退出 | “训练中断了。常见原因是数据太少或内存不足，可减少数据或换更小的模型再试。” |
| 显存/内存不足（OOM 关键字） | “内存不够用了。建议用更小的模型，或减少一次训练的数据量。” |
| 推理加载失败 | “模型加载失败，请确认训练已正常完成。” |

真实报错记录在 `job.error`（给开发看），`job.message` 给小白看。

## 7. 默认参数（小白安全值）

| 参数 | 默认 | 说明 |
|---|---|---|
| epochs | 3 | 比 M0 的 30 保守，控制耗时 |
| learning_rate | 2e-4 | 沿用 |
| lora_rank | 8 | 沿用 |
| batch_size | 2 | 16GB 安全 |
| 精度 | MPS=fp32 | M0 验证结论 |

数据量小（几十条）+ 3 epoch，单次训练预计几分钟内，符合小白可接受体验。

## 8. 测试计划

| 测试 | 类型 | 说明 |
|---|---|---|
| 精度选择 | 单元 | MPS/CPU→fp32，CUDA→bf16 |
| trainer_log 解析 | 单元 | 给定样例 jsonl，正确算出 progress/loss/eta |
| 引擎工厂 | 单元 | 环境就绪→真实引擎；否则 Mock |
| 配置生成不再写死 bf16 | 单元 | 扩展现有 `test_llamafactory_config` |
| 真实训练 smoke | 手动 | 用极小数据 + 1 epoch 跑通真实链路（验证用，不进 CI） |

CI 友好：所有自动化测试不依赖真实模型权重；真实训练只做手动 smoke。

2. `llamafactory.py`（去 bf16 写死）
3. `infer.py`（真实对比）

## 10. 风险

| 风险 | 缓解 |
|---|---|
| 真实推理慢，阻塞事件循环 | 放线程池/子进程执行 |
| 子进程终止不干净（残留 llamafactory 进程） | terminate→超时 kill，记录 pid |
| 不同 Mac 内存差异导致 OOM | 默认小模型 + 保守参数 + OOM 提示 |
| 真实模型未下载导致整链路不可用 | 环境检测 + 自动降级 Mock + 明确告知用户当前模式 |

## 11. 实现结果（2026-06-18 已完成）

本设计已全部实现并通过端到端真实验证。

### 11.1 实现内容

| 模块 | 改动 | 状态 |
|---|---|---|
| `hardware.py` | 新增 `detect_device` / `select_precision` / `torch_dtype` / `training_env` / `llamafactory_cli`（按 venv 解析 CLI 绝对路径） | ✅ |
| `llamafactory.py` | 精度不再写死 bf16，按设备选 fp32/bf16；`system` 列仅在数据含 system 时声明，避免 `KeyError: 'system'` | ✅ |
| `infer.py` | base vs base+LoRA 真实推理，带 CLI 可被子进程调用 | ✅ |
| `engine.py` | `TrainingEngine` 协议 + `LlamaFactoryTrainingEngine`（真实）+ `MockTrainingEngine`（降级）+ `parse_trainer_log` + `build_engine` 工厂 | ✅ |
| `main.py` | 启动时工厂选引擎；`/api/environment` 暴露 `engine`/`engine_label`；导出改 `FileResponse` 返回 zip | ✅ |
| `domain.py` | `TrainingJob` 增加 `engine`/`pid`/`run_dir`/`eta` | ✅ |
| `web/app.js` | 环境卡片显示「真实训练 / 演示模式」 | ✅ |
| `tests/` | 新增 trainer_log 解析、精度策略、引擎工厂测试 | ✅ |

### 11.2 验证结果

- **真实训练 smoke**：`LlamaFactoryTrainingEngine` 成功启动 `llamafactory-cli` 子进程，Qwen2.5-0.5B + LoRA 训练跑到 100%，产出 `adapter_model.safetensors`。
- **进度回填**：从 `trainer_log.jsonl` 实时解析出 progress/loss/eta，job 状态正确流转到 `completed`。
- **推理对比**：`infer.py` 子进程加载 base 与 base+LoRA，返回 before/after 双回答（机制打通；回答质量取决于真实数据量，smoke 用 8 条重复样本仅验证链路）。
- **导出**：`export()` 把 output 目录打包为 `lora-adapter-{job_id}.zip`，含 adapter + checkpoint。
- **质量门禁**：11 个单测全过，`ruff check` 通过。

### 11.3 踩坑记录（供后续复用）

1. **`llamafactory-cli` 不在 PATH**：venv 未激活时子进程找不到命令。改为按 `sys.executable` 旁路解析 CLI 绝对路径。
2. **`KeyError: 'system'`**：`dataset_info.json` 声明了 system 列，但 `exclude_none` 让无 system 的记录缺该键，LLaMA-Factory 崩溃。改为按数据是否含 system 动态决定列声明与写入。
