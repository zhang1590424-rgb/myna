# 本地小白模型微调工具 — 设计策划案

> 日期:2026-06-18
> 状态:已与产品负责人对齐,待评审

## 一、产品定位与核心价值

**一句话定位**:让完全不懂技术的人,在自己的 Mac 上,第一次成功把一个开源小模型「调教」成自己想要的样子。

**核心价值(啊哈时刻)**:用户输入自己的数据 → 点几下 → 训练完看到「同一句话,训练前模型这么答、训练后那么答」的明显变化。要交付的唯一核心感受是:**「我居然真的训练出了一个 AI」**。

**目标用户**:对模型训练完全不懂的小白(典型如产品经理、运营、个人开发者等),使用 Mac(Apple M 系列芯片)电脑。

### 与 LLaMA-Factory 的关系(参考项目结论)

- **不基于其源码做魔改,而是把其训练引擎当作底层依赖来调用。**
- 原因:LLaMA-Factory 的 GUI(LLaMA Board)本质是「参数表单」,面向算法工程师;本工具做的是「引导式产品层」,面向小白。两者设计哲学相反,改源码不如在外面包一层干净。
- 路线原则:**底层引擎能复用的尽量复用,不重复造轮子;交互层完全重做。**
- 类比:LLaMA-Factory 是「专业相机的全手动模式」,本工具是「手机相机的傻瓜一键拍」——成像引擎可复用,交互彻底重做。

## 二、核心决策回顾

| # | 维度 | 结论 |
|---|---|---|
| 1 | 核心目标 | 面向小白的本地后训练工具,基于开源小模型 |
| 2 | 目标机器 | Mac(Apple M 系列芯片) |
| 3 | 训练能力 | 第一版只做 SFT(LoRA 监督微调),强化学习(RL)留作后续规划 |
| 4 | 场景 | 通用,但配预制模板引导 + 可自定义 |
| 5 | 形态 | 桌面应用(双击即用) |
| 6 | 效果验证 | 训练前后对话对比(主) + 折叠的专业指标 |

> 说明:RL(DPO/PPO 等)在 Mac 本地基本不可行——RL 训练需同时驻留多个模型(策略/参考/奖励),显存与算力开销远高于 SFT,且主流 RL 框架依赖 CUDA,与 Mac MPS 生态不兼容。故第一版聚焦 SFT。

## 三、用户核心流程(小白视角)

整个产品是一条「五步直线」,不让小白有岔路。每步只问一件事,默认值预填,可一路「下一步」到底。

```
① 选方向  →  ② 喂数据  →  ③ 确认设置  →  ④ 看着它训练  →  ⑤ 试用对比
```

| 步骤 | 小白看到什么 | 背后发生什么 | 防呆设计 |
|---|---|---|---|
| ① 选方向 | 卡片:客服话术 / 角色扮演 / 文本改写 / 知识问答 / 自定义 | 每个模板预设:推荐基础模型 + 数据格式 + 训练参数 | 选模板=自动配好 80% 的东西 |
| ② 喂数据 | 上传数据 + 可下载的示例数据(可直接参考/试用) | 自动校验格式、转成训练格式、统计条数 | 格式错给人话提示,不报红色英文 |
| ③ 确认设置 | 一句话总结:「将基于 XX 模型,用你的 N 条数据训练」+ 折叠的高级选项 | 高级参数(学习率/轮数等)全部预设、折叠隐藏 | 小白无需碰参数 |
| ④ 看着它训练 | 进度条 + 大白话状态(「正在学习第 3 轮…」)+ 折叠 loss 曲线 | 调用引擎跑 LoRA,实时回传进度 | 可中途停止;失败给人话原因 |
| ⑤ 试用对比 | 左右分屏:原模型 vs 你的模型,输入同一句话对比 | 同时加载两个模型推理 | 一眼看到「学会了」 |

### 关键产品决策

- **基础模型来源**:工具内置一份「Mac 友好」精选清单(不同体量,如 0.5B / 1.5B / 3B 量级),按机器配置给出推荐标记,首次使用时自动下载,小白无需了解「去哪下模型」。
- **训练产物**:支持「在线试用」+「导出模型」两者,既能即时看效果,也能带走成果。

## 四、技术架构

整体为「外壳 App + 本地引擎」结构:

```
┌─────────────────────────────────────────┐
│   桌面 App 外壳(用户双击打开的东西)        │
│   - 引导界面(选方向/喂数据/试用)          │
│   - 大白话状态、防呆提示                   │
└───────────────┬─────────────────────────┘
                │ 本地通信(用户无感)
┌───────────────▼─────────────────────────┐
│   本地服务(藏在 App 里,自动启动)         │
│   - 调用 LLaMA-Factory 引擎做 SFT/推理     │
│   - 管数据转换、模型下载、训练任务         │
└───────────────┬─────────────────────────┘
                │
┌───────────────▼─────────────────────────┐
│   底层引擎:LLaMA-Factory + PyTorch(MPS)  │
│   - 真正干训练活的部分,复用现成能力       │
└─────────────────────────────────────────┘
```

### 技术选择与理由

| 选择 | 方案 | 理由 |
|---|---|---|
| 训练引擎 | 复用 LLaMA-Factory(命令行/Python 调用) | 不重复造轮子 |
| 计算后端 | PyTorch MPS(Apple 芯片 GPU 加速) | Mac 上唯一能用 GPU 的路子 |
| 外壳 App | Tauri / Electron(网页技术做界面,包成 Mac App) | 界面美观灵活、跨平台、未来出 Windows 版容易;延续「本地网页→桌面 App」的演进路线 |
| 本地服务 | Python(与引擎同语言) | 和引擎同语言,集成最顺 |

### 模块边界(便于独立理解与测试)

- **外壳 App(前端)**:只负责界面与引导,通过本地接口与后端通信,不含训练逻辑。
- **本地服务(后端)**:负责数据转换、模型下载、训练任务调度、推理服务;对前端暴露清晰接口。
- **引擎适配层**:封装对 LLaMA-Factory 的调用(参数拼装、进度解析),使上层不直接耦合引擎细节,便于后续替换或升级引擎。

### 风险与对用户的影响(如实标注)

1. **首次使用要下载数 GB**(Python 环境 + 模型文件):首次启动等待较久。
   - 缓解:清晰的下载进度 + 「正在准备,仅首次需要」的安抚文案。
2. **Mac 训练速度有限**:M 系列芯片训练比专业 GPU 慢,小模型 + 几百条数据可能需几分钟到几十分钟。
   - 缓解:预制模板控制数据量与模型大小,把单次训练控制在可接受时间内。
3. **打包复杂度高(最大工程难点)**:把 Python 训练环境塞进双击即用的 Mac App(环境隔离、签名、体积)。若做不好可能「装了打不开」。
   - 缓解:作为第一版最优先验证的技术点(见里程碑 M2)。

## 五、范围边界(第一版)

### ✅ 第一版做(MVP)

- SFT(LoRA 监督微调)一种训练方式
- 五步直线流程:选方向 → 喂数据 → 确认 → 训练 → 试用对比
- 4 个预制模板(客服 / 角色扮演 / 文本改写 / 知识问答)+ 自定义入口
- 内置 Mac 友好小模型清单(0.5B / 1.5B / 3B 量级),自动下载、按配置推荐
- 数据格式校验 + 示例数据下载
- 训练前后「对话对比」+ 折叠的 loss 曲线
- 训练产物在线试用 + 导出
- Mac 桌面 App(双击即用)

### ❌ 第一版不做(后续规划)

- 强化学习(DPO / PPO 等)—— Mac 跑不动,留到支持云端再说
- 云端 GPU 训练
- Windows / Linux 版(架构留余地,但不在第一版交付)
- 多人协作 / 账号体系 / 云端存储
- 全参数微调(只做 LoRA)
- 复杂的数据标注 / 清洗工具(只做基础校验)

## 六、开发里程碑(按风险优先排序)

不按「先界面后后端」,而是先攻最大风险点,确保路走得通再投入做漂亮。

| 阶段 | 目标 | 验证什么 |
|---|---|---|
| M0:技术验证(最关键) | 命令行跑通:Mac 上用 LLaMA-Factory + MPS,把小模型 LoRA 微调出效果 | 证明「Mac 本地能训练出有效果的模型」根本成立 |
| M1:核心流程跑通 | 最简界面串起「喂数据→训练→试用对比」全流程 | 证明产品闭环成立 |
| M2:打包验证 | 把 Python 环境 + 引擎打成能双击运行的 Mac App | 攻克最大工程难点:打包 |
| M3:产品化 | 加预制模板、防呆引导、美化界面、模型自动下载、导出 | 做成真正给小白用的样子 |
| M4:小白实测 | 找真·小白试用,看能否独立走完全程 | 验证「小白真的会用」 |

**核心逻辑**:M0 和 M2 是两个最可能翻车的点(能否训出效果、能否打包成 App),放最前面验证,避免界面做完才发现底层走不通。

## 七、交互设计与视觉规范

已通过可点击高保真原型确认,原型文件:`prototype/interaction-prototype.html`。

### 7.1 界面结构

- **左侧步骤栏(Step Rail)**:固定展示 0~5 共 6 个步骤,标记当前步与已完成步(已完成打勾),让用户随时知道"在哪、还剩几步",消除未知焦虑。
- **中间主面板**:一次只呈现当前步骤的内容,大留白,单任务聚焦。
- **底部操作栏**:左"上一步"(弱化),右"下一步/开始训练"(唯一强调按钮)。第 5 步出现"导出模型"。

### 7.2 五步 + 准备页内容

| 步骤 | 标题 | 关键元素 |
|---|---|---|
| 0 准备环境 | 正在为你准备训练环境 | 首次下载进度(约 2.4 GB)+ "仅首次需要"安抚文案 |
| 1 选方向 | 你想教 AI 学会做什么 | 5 张卡片:客服话术 / 角色扮演 / 文本改写 / 知识问答 / 自定义 |
| 2 喂数据 | 把你的数据喂给它 | 拖拽上传 + 格式校验结果(人话)+ 示例数据下载 |
| 3 确认设置 | 确认一下就开始 | 一句话总结 + 折叠的高级参数 |
| 4 开始训练 | 正在训练你的 AI | 大号百分比 + 大白话状态 + 折叠 loss 曲线 |
| 5 试用对比 | 同一个问题看前后差别 | 训练前/后左右并排 + 输入框对比 + 导出 |

### 7.3 视觉风格:克制单色(Codex 风)

定调来源:用户明确偏好 Codex 那种高级、简洁、克制的观感,放弃彩色糖果风。

| 维度 | 规范 |
|---|---|
| 色彩策略 | Restrained(克制):近乎纯灰阶 + 唯一近黑强调色,几乎无彩色 |
| 唯一强调色 | 近黑 `oklch(0.27 0.005 280)`,仅用于主按钮 / 当前步骤序号 / 进度条 |
| 中性色 | 全部 OKLCH 灰阶,微微偏冷(hue 280),不用纯黑 `#000` / 纯白 `#fff` |
| 字体 | 系统中性无衬线(-apple-system / SF Pro / PingFang SC),不用圆体 |
| 图标 | 线性单色,描边 1.5px,无彩色底块 |
| 层级手段 | 靠留白、字重、灰阶层次拉开,而非颜色 |
| 状态色 | 成功用低饱和青灰 `oklch(0.55 0.02 175)`,克制使用 |
| 圆角 | 卡片 14px、小元素 10px,统一温和 |
| 动效 | 仅 `translateY` 微入场,ease-out 曲线;尊重 `prefers-reduced-motion` |

### 7.4 设计硬性原则(遵循 impeccable 禁令)

- 唯一强调按钮:全局只有一个强调色按钮,用户永远知道"下一步点哪"。
- 大白话:状态文案不出现术语(如用"正在学习第 2 轮"而非"epoch 2/3")。
- 专业信息折叠:高级参数、loss 曲线默认收起,不吓到小白、不挡住进阶用户。
- 禁用:侧边色条、渐变文字、默认玻璃拟态、多强调色、文案破折号。
