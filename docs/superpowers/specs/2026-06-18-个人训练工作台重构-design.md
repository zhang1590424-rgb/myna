# 个人训练工作台重构设计

## 背景

当前「小白训练师」定位是一次性体验工具——帮完全不懂的人成功训练一次。现在需要转型为**个人训练工作台**，服务于长期模型训练学习：反复实验、积累认知、逐步进阶。

用户：老张，Mac M2/16GB，产品经理，想通过动手实验建立对模型训练的深入理解。

## 设计原则

1. **工作台，不是教程**：工具不教你，给你实验能力让你自己探索
2. **资源 + 组合实验**：模型/数据独立管理，实验是它们的组合
3. **历史即资产**：每次实验自动存档，长期积累可回溯可对比
4. **硬件务实**：所有功能在 Mac M2/16GB 上可用，不追求跑大模型

## 信息架构

四大模块：

```
模型管理  │  数据管理  │  实验（含参数配置）│  实验室（对话验证）
```

### 模块职责

| 模块 | 管理什么 | 核心操作 |
|---|---|---|
| **模型管理** | 本地底座模型 | 查看/下载/删除，查看模型信息 |
| **数据管理** | 训练数据集 | 上传 CSV/JSON/JSONL，预览，管理多份数据 |
| **实验** | 训练运行 = 模型 + 数据 + 参数 → 产出 | 新建、克隆、对比、查看历史 |
| **实验室** | 和训练产出对话验证 | 加载 adapter，发 prompt 看效果 |

### 模块关系

```
模型 ──┐
       ├──→ 实验 ──→ adapter 产出 ──→ 实验室（对话验证）
数据 ──┘
       │
    参数（内嵌在实验中，不独立管理）
```

## 模型清单

4 个模型，覆盖不同实验维度：

| 模型 | 参数量 | 学习价值 | 来源 |
|---|---|---|---|
| Qwen2.5-0.5B-Instruct | 0.5B | 试错成本最低的"草稿纸" | ModelScope |
| Qwen2.5-1.5B-Instruct | 1.5B | 效果与速度平衡，日常主力 | ModelScope |
| Gemma-2-2B-it | 2B | 不同架构（Google系），横向对比 | ModelScope/HF |
| DeepSeek-R1-Distill-Qwen-1.5B | 1.5B | 推理增强蒸馏，体验特化底座再 SFT | ModelScope |

模型清单从代码硬编码改为配置文件（`local_trainer/model_registry.yaml`），后续加模型改配置即可。

## 训练方法

### SFT（监督微调）

现有功能，数据格式不变（instruction/input/output 或 question/answer）。

### DPO（偏好优化）

新增。数据格式：

```json
{
  "instruction": "用户的问题",
  "chosen": "偏好的回答",
  "rejected": "不想要的回答"
}
```

实现方式：LLaMA-Factory 原生支持 DPO stage，配置生成切换 `stage: dpo` + `pref_beta` 参数。

使用场景：SFT 训出来"方向对但风格不满意"时，用 DPO 精调偏好。

## 数据模型

### Experiment（实验）

```
id: string (UUID)
name: string（自动生成，如 "sft-qwen0.5b-客服-#3"，可改）
status: pending | queued | running | completed | failed | stopped
method: "sft" | "dpo"
model_id: string
dataset_id: string
params: JSON { epochs, learning_rate, lora_rank, batch_size, [dpo: beta] }
cloned_from: string | null
metrics: JSON { final_loss, peak_loss, duration_sec, steps_total, ... }
output_dir: string
notes: string（自由笔记）
tags: string[]
created_at: datetime
started_at: datetime | null
completed_at: datetime | null
```

### Dataset（数据集）

```
id: string (UUID)
name: string
source_filename: string
format: "alpaca" | "dpo_pairs"
row_count: int
file_path: string
created_at: datetime
```

### Model（本地模型）

```
id: string
name: string
repo_id: string
parameter_count: string
local_path: string | null
available: bool
download_size_label: string
```

### 存储

SQLite（`runtime/workbench.db`）：
- 单文件，无依赖，Python 标准库自带
- 支持查询筛选（按模型/方法/日期）
- 备份 = 复制一个文件

## 核心工作流

### 1. 新建实验

一页完成：选模型 → 选数据集 → 选方法(SFT/DPO) → 填参数（默认值预填，可展开调整）→ 命名(可选) → 开始。

### 2. 克隆 & 变量实验

从实验列表选一条 → "以此为模板新建" → 预填全部配置 → 修改想变的变量（高亮变化项）→ 开始。

批量变量：选定一个维度（如 epochs），设置多个值（1, 3, 5），一次生成多个实验排队执行。

### 3. 实验对比

勾选 2-3 个实验 → "对比" → 进入对比视图：
- 参数差异表（只显示不同项，相同的折叠）
- Loss 曲线叠加（同一张图多条线）
- 输出对比（同一 prompt，各实验模型的回答并排）

### 4. 实验室（对话验证）

选一个已完成实验 → "试用" → 加载 base model + LoRA adapter → 对话界面。

功能：
- 单轮对话验证
- 保存"测试 prompt 集"——常用检验问题，一键跑完看效果
- 切换 adapter（卸载当前 → 加载另一个实验的）

### 5. 实验排队

创建多个实验后按序执行：
- 同一时间只有一个在 running（Mac 硬件限制）
- pending 实验按创建顺序排队
- 可晚上配好多个实验挂着跑，第二天看结果

### 6. 实验笔记

每个实验详情页有自由文本区，跑完随手记"这次学到什么"。

```
│  前端（web/）— 纯静态 SPA，4 个视图                   │

└────────────────────────┬────────────────────────────┘
                         │ HTTP /api/*
┌────────────────────────▼────────────────────────────┐
│  后端（local_trainer/）— FastAPI                     │
│                                                     │
│  model_registry    — 模型清单（YAML驱动）+ 下载      │
│  dataset_manager   — 数据集 CRUD + 格式校验          │
│  experiment_service — 实验 CRUD/克隆/状态管理         │
│  queue_manager     — 排队调度，按序启动训练           │

│  inference_engine  — 实验室推理（重构自 infer.py）    │

│                                                     │
└────────────────────────┬────────────────────────────┘
                         │
              ┌──────────▼──────────┐
              │  LLaMA-Factory CLI  │
              │  (训练 & 推理引擎)   │
              └─────────────────────┘
```

### 前端技术选型

保持纯静态（HTML + CSS + vanilla JS），不引入框架：
- 个人工具，不需要组件化复杂度
- 零构建依赖，改完刷新即生效
- 当前 app.js 有路由基础，扩展为 4 视图可行

若后续复杂度明显上升再考虑轻量框架（如 Alpine.js 或 Preact）。

### 后端模块划分

| 新模块 | 复用现有 | 说明 |
|---|---|---|
| `model_registry.py` | 替换 `templates.get_model_catalog()` | YAML 驱动，支持 4 个模型 |
| `dataset_manager.py` | 复用 `data_validation.py` + `store.py` | 加 SQLite 持久化 |
| `experiment_service.py` | 替换 `JobStore` | 完整实验生命周期 |
| `queue_manager.py` | 新增 | 排队调度 |
| `inference_engine.py` | 重构自 `infer.py` | 独立推理服务 |
| `persistence.py` | 新增 | SQLite 数据访问层 |
| `compare.py` | 新增 | 对比逻辑 |

### API 设计概要

```
# 模型
GET    /api/models                     — 模型清单
POST   /api/models/{id}/download       — 下载模型
DELETE /api/models/{id}                 — 删除本地模型

# 数据集
GET    /api/datasets                    — 数据集列表
POST   /api/datasets                    — 上传新数据集
GET    /api/datasets/{id}               — 数据集详情 + 预览
DELETE /api/datasets/{id}               — 删除数据集

# 实验
GET    /api/experiments                 — 实验列表（支持筛选）
POST   /api/experiments                 — 新建实验
POST   /api/experiments/{id}/clone      — 克隆实验
GET    /api/experiments/{id}            — 实验详情（含 metrics）
PATCH  /api/experiments/{id}            — 更新笔记/标签/名称
POST   /api/experiments/{id}/stop       — 停止训练
DELETE /api/experiments/{id}            — 删除实验
POST   /api/experiments/batch           — 批量创建（变量实验）
GET    /api/experiments/compare?ids=... — 对比数据

# 实验室
POST   /api/lab/load                    — 加载某实验的 adapter
POST   /api/lab/chat                    — 发送 prompt 获取回答
POST   /api/lab/unload                  — 卸载当前模型
GET    /api/lab/status                  — 当前加载状态

# 队列
GET    /api/queue                       — 队列状态
POST   /api/queue/pause                 — 暂停队列
POST   /api/queue/resume                — 恢复队列
```

## 实现顺序（一次性交付，按依赖排序）

1. **SQLite 持久化层** — 建表、CRUD、迁移
2. **模型注册表** — YAML 配置 + registry 模块 + 下载适配
3. **数据集管理** — 重构 store，支持多数据集持久化 + DPO 格式
4. **实验服务** — 替换 JobStore，完整生命周期 + 克隆 + 笔记
5. **排队管理** — 实验排队调度
6. **DPO 配置生成** — 扩展 llamafactory.py 支持 DPO stage
7. **推理引擎** — 重构 infer.py 为独立可控的推理服务
8. **对比逻辑** — metrics 聚合 + 差异计算
9. **前端重写** — 四视图 SPA + 所有交互
10. **API 路由** — 新增所有端点，替换旧的五步向导 API
11. **集成测试** — 端到端验证

## 迁移策略

- 现有五步向导的代码不删除，但不再作为默认入口
- `main.py` 的旧 API 逐步替换为新 API
- `runtime/workbench.db` 新建，不需要从旧数据迁移（之前是内存，没持久化）
- 前端从零重写（当前 app.js 是向导逻辑，工作台逻辑完全不同）

## 硬件约束与保护

- 同一时间最多运行 1 个训练任务
- 推理加载最多 1 个模型（base + adapter）
- 训练时不能同时推理，推理时不能同时训练（内存互斥）
- 启动训练前检查内存是否足够，不足给人话提示

## 风险点

1. **Gemma 模型兼容性**：LLaMA-Factory 对 Gemma 的支持需验证，可能需要特定 template 配置
2. **DPO 数据准备门槛**：偏好数据需要 chosen/rejected 对，比 SFT 数据更难准备——工具层面暂不解决，靠用户手动构造
3. **SQLite 并发**：单用户场景不是问题，但前端轮询 + 后台训练同时写 DB 需要 WAL 模式
4. **前端复杂度**：四视图 + 对比 + 对话，纯 vanilla JS 可能到后期比较痛苦——预留切框架的空间
